# 📘 1. STRING vs STRINGBUILDER vs STRINGBUFFER

---

# 🔥 String

```java
String s = "Java";
```

## Characteristics

* Immutable
* Stored in String Constant Pool (SCP)
* Every modification creates new object

---

## Example

```java
String s = "Java";
s.concat(" Tech");

System.out.println(s);
```

✅ Output:

```text
Java
```

Because String is immutable.

---

# 🔥 StringBuilder

```java
StringBuilder sb = new StringBuilder("Java");
```

## Characteristics

* Mutable
* Faster
* Not thread-safe
* Introduced in Java 1.5

---

# 🔥 Why mutable?

```java
sb.append(" Tech");
```

Modifies same object.

---

# 🔥 Important Methods

---

## append()

```java
StringBuilder sb = new StringBuilder("Java");
sb.append(" Tech");

System.out.println(sb);
```

✅ Output:

```text
Java Tech
```

---

## insert()

```java
sb.insert(4, "123");
```

---

## replace()

```java
sb.replace(0, 4, "Python");
```

---

## delete()

```java
sb.delete(0, 2);
```

---

## reverse()

```java
sb.reverse();
```

---

## capacity()

Default capacity:

```text
16
```

Formula when expanded:

```text
(newCapacity = oldCapacity * 2 + 2)
```

---

## ensureCapacity()

```java
sb.ensureCapacity(100);
```

---

## trimToSize()

Removes extra unused capacity.

---

# 🔥 StringBuffer

Almost same as StringBuilder except:

✅ Thread-safe
❌ Slower

---

# 🔥 Why thread-safe?

Because methods are synchronized.

Example internally:

```java
public synchronized StringBuffer append(String str)
```

---

# 🔥 Important Difference

| Feature     | StringBuilder | StringBuffer |
| ----------- | ------------- | ------------ |
| Thread-safe | ❌             | ✅            |
| Speed       | Faster        | Slower       |

---

# 🔥 equals() behavior

```java
StringBuilder sb1 = new StringBuilder("Java");
StringBuilder sb2 = new StringBuilder("Java");

System.out.println(sb1.equals(sb2));
```

✅ Output:

```text
false
```

---

# 🔥 Why?

Because StringBuilder does NOT override:

```java
Object.equals()
```

So:

```text
reference comparison happens
```

---

# 🔥 Compare with String

```java
String s1 = "Java";
String s2 = "Java";

System.out.println(s1.equals(s2));
```

✅ Output:

```text
true
```

Because String overrides equals().

---

# 🔥 StringBuilder is final

```java
public final class StringBuilder
```

---

# 🔥 Why final?

* Security
* Prevent incorrect inheritance
* JVM optimizations
* Maintain mutable behavior safely

---

# 📘 2. FINAL KEYWORD

---

# 🔥 final variable

```java
final int x = 10;
```

Cannot reassign.

---

# 🔥 final method

```java
final void show() {}
```

Cannot override.

---

# 🔥 final class

```java
final class A {}
```

Cannot inherit.

---

# 🔥 Why use final?

* Security
* Immutability
* Optimization
* Prevent misuse

# 📘 3. INHERITANCE

---

# 🔥 Definition

Inheritance means:

```text
one class acquires properties and behavior of another
```

---

# 🔥 Syntax

```java
class Child extends Parent
```

---

# 🔥 Types

| Type             | Allowed? |
| ---------------- | -------- |
| Single           | ✅        |
| Multilevel       | ✅        |
| Hierarchical     | ✅        |
| Multiple (class) | ❌        |

---

# 🔥 IS-A Relationship

```text
Dog IS-A Animal
```

---

# 🔥 Constructor Flow

```java
class A {
    A() {
        System.out.println("A");
    }
}

class B extends A {
    B() {
        System.out.println("B");
    }
}
```

```java
new B();
```

✅ Output:

```text
A
B
```

---

# 🔥 Why?

Parent constructor executes first.

---

# 🔥 Overriding

```java
class A {
    void show() {
        System.out.println("A");
    }
}

class B extends A {

    @Override
    void show() {
        System.out.println("B");
    }
}
```

---

# 🔥 @Override mandatory?

❌ No

But recommended.

---

# 🔥 Why recommended?

Compiler checks mistakes.

---

## Tricky Example

```java
class A {
    void show() {}
}

class B extends A {

    @Override
    void show(int x) {}
}
```

❌ Compile-time error.

Without @Override:
✅ No error but method overloaded accidentally.

---

# 🔥 Variable Hiding

```java
class Parent {
    int age = 58;

    void print() {
        System.out.println(age);
    }
}

class Child extends Parent {
    int age = 20;
}
```

```java
Child c = new Child();
c.print();
```

✅ Output:

```text
58
```

---

# 🔥 Why?

Because:

```text
variables are NOT overridden
```

Method belongs to Parent.

So Parent.age used.

---

# 🔥 Key Rule

| Feature  | Methods    | Variables    |
| -------- | ---------- | ------------ |
| Binding  | Runtime    | Compile-time |
| Behavior | Overriding | Hiding       |

---

# 🔥 Static in inheritance

---

## Static method hiding

```java
class Parent {
    static void show() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    static void show() {
        System.out.println("Child");
    }
}
```

```java
Parent p = new Child();
p.show();
```

✅ Output:

```text
Parent
```

---

# 🔥 Why?

Static methods use:

```text
compile-time binding
```

---

# 🔥 Important

Static methods are:

```text
hidden, not overridden
```

---

# 🔥 Meaning of hidden

Child creates another method with same name.

No runtime polymorphism.

---

# 🔥 super keyword

```java
super.show();
```

Calls parent version.

# 📘 4. INTERFACE

---

# 🔥 Definition

Interface = contract.

Defines:

```text
WHAT to do
```

not:

```text
HOW to do
```

---

# 🔥 Example

```java
interface Animal {
    void sound();
}
```

---

# 🔥 Implementing

```java
class Dog implements Animal {

    public void sound() {
        System.out.println("Bark");
    }
}
```

---

# 🔥 Important defaults

---

## Methods

```java
void show();
```

becomes:

```java
public abstract void show();
```

---

## Variables

```java
int x = 10;
```

becomes:

```java
public static final int x = 10;
```

---

# 🔥 Types of interface methods

---

## Abstract method

```java
void show();
```

---

## Default method

```java
default void show() {}
```

---

## Static method

```java
static void test() {}
```

---

## Private method (Java 9+)

```java
private void helper() {}
```

---

# 🔥 Interface static methods

---

## Important Rule

```text
Interface static methods are NOT inherited
```

---

# 🔥 Example

```java
interface A {

    static void show() {
        System.out.println("A");
    }
}

class B implements A {

    static void show() {
        System.out.println("B");
    }
}
```

---

# 🔥 Is this overriding?

❌ NO

Because static methods are not inherited.

---

# 🔥 What happens?

B creates:

```text
its own fresh static method
```

---

# 🔥 Proof

```java
A.show();
B.show();
```

Both work independently.

---

# 🔥 Even interface extending interface

```java
interface A {
    static void show() {}
}

interface B extends A {
    static void show() {}
}
```

Still:
❌ NOT overriding

---

# 🔥 Why?

Static methods are:

```text
class/interface level
```

not object-level.

---

# 🔥 Multiple inheritance via interface

```java
interface A {}
interface B {}

class C implements A, B {}
```

✅ Allowed

---

# 🔥 Default method conflict

```java
interface A {
    default void show() {}
}

interface B {
    default void show() {}
}
```

```java
class C implements A, B {

    public void show() {}
}
```

Must override.

---

# 🔥 Main Areas

| Area        | Stores                      |
| ----------- | --------------------------- |
| Stack       | local vars, references      |
| Heap        | objects                     |
| Method Area | class metadata, static data |

---

# 🔥 Local variables

```java
void test() {
    int x = 10;
}
```

Stored in:

```text
Stack
```

---

# 🔥 Objects

```java
new Student();
```

Stored in:

```text
Heap
```

---

# 🔥 Instance variables

Stored inside object in:

```text
Heap
```

---

# 🔥 Static variables

Stored in:

```text
Method Area / Metaspace
```

---

# 🔥 Methods

Bytecode stored in:

```text
Method Area
```

---

# 🔥 String literals

```java
String s = "Java";
```

Stored in:

```text
String Constant Pool
```

inside Heap.

---

# 🔥 Interface variables in JVM

```java
interface A {
    int x = 10;
}
```

Since:

```java
public static final
```

stored in:

```text
Method Area
```

---

# 🔥 Interface methods in JVM

| Method Type             | Stored Where |
| ----------------------- | ------------ |
| Abstract metadata       | Method Area  |
| Default method bytecode | Method Area  |
| Static method bytecode  | Method Area  |

# 📘 6. POLYMORPHISM

---

# 🔥 Definition

```text
One thing taking many forms
```

---

# 🔥 Types

| Type         | Example     |
| ------------ | ----------- |
| Compile-time | Overloading |
| Runtime      | Overriding  |

---

# 🔥 Compile-time polymorphism

```java
void add(int a, int b)
void add(int a, int b, int c)
```

Compiler decides method.

---

# 🔥 Runtime polymorphism

```java
Animal a = new Dog();
a.sound();
```

JVM decides method at runtime.

---

# 🔥 Why runtime?

Because actual object is:

```text
Dog
```

---

# 🔥 Upcasting

```java
Animal a = new Dog();
```

✅ Valid

---

# 🔥 Why valid?

Dog IS-A Animal.

---

# 🔥 Downcasting

```java
Dog d = (Dog) a;
```

Works only if actual object is Dog.

---

# 🔥 Invalid

```java
Dog d = new Animal();
```

❌ Compile-time error

---

# 🔥 Why?

Every Animal is not Dog.

---

# 🔥 Another invalid

```java
Animal a = new Animal();
Dog d = (Dog) a;
```

❌ Runtime:

```text
ClassCastException
```

---

# 🔥 Variables and polymorphism

```java
class A {
    int x = 10;
}

class B extends A {
    int x = 20;
}
```

```java
A obj = new B();
System.out.println(obj.x);
```

✅ Output:

```text
10
```

No runtime polymorphism for variables.

---

# 🔥 Static methods and polymorphism

```java
A obj = new B();
obj.show();
```

Static methods:
❌ No runtime polymorphism.

---

# 🔥 Dynamic Method Dispatch

For overridden methods:

```text
JVM checks actual object at runtime
```
